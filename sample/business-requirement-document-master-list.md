# 参考: 要求定義として必要な文書の一覧

必要文書一覧 最終版

## 0. 共通運用ルール

この最終版では、各文書クラスごとに次のメタ情報を必須にします。

* 文書クラスID
* 文書名
* 必須度: **Core / Conditional / Optional**
* 状態: **Confirmed / Tentative / Unknown**
* SoT/出典
* オーナー
* 対象リリース
* 対象環境
* 対象地域/法人/テナント
* Prompt投入可否
* 最終確認日
* 関連 ADR / 未解決論点

これにより、後でユーザー保有文書を「どこに該当するか」で機械的にマッピングできます。

## 1. 必要文書クラス一覧

### D01. 事業意図・成功条件定義書

**必須度:** Core
**目的:** なぜ作るのか、何を成功とみなすかを固定する。
**最低内容:** 経営課題、開発の必要性、放置コスト、期待価値、KPI現状/目標、成功条件、失敗条件、今回やらないこと、優先順位の理由、ROI仮説。
**元台帳対応:** A
**該当しやすい既存文書名:** 企画書、事業計画、案件起案書、投資対効果試算、スポンサー説明資料。
**不足判定:** KPI と成功条件がない場合は不足。A-02 と A-14 は分離し、前者は「必要性」、後者は「今やる理由」にする。

### D02. スコープ・対象境界定義書

**必須度:** Core
**目的:** どこまで作り、どこを触らないかを確定する。
**最低内容:** 対象事業、対象業務、対象国/地域/法人/テナント、対象チャネル/ユーザー/デバイス、開発種別、対象外、変更禁止領域、他案件依存、先行前提、対象環境。
**元台帳対応:** B
**該当しやすい既存文書名:** スコープ定義書、対象範囲表、境界図、フェーズ計画。
**不足判定:** Out of scope、変更禁止、依存条件の3つが欠けていれば不足。B-12 は案件全体区分、B-13 は機能単位区分に分ける。

### D03. ステークホルダー・承認権限・責任分担表

**必須度:** Core
**目的:** 誰が決め、誰が責任を負い、誰が例外を許可するかを固定する。
**最低内容:** スポンサー、事業責任者、プロダクト責任者、業務責任者、データオーナー、システムオーナー、セキュリティ/法務/監査責任者、承認権限表、RACI、エスカレーション経路、例外承認者、緊急変更承認者。
**元台帳対応:** C
**該当しやすい既存文書名:** RACI表、承認マトリクス、責任分担表、会議体定義。
**不足判定:** C-13 と C-14 を分けて持っていなければ不足。

### D04. 業務プロセス仕様書

**必須度:** Core
**目的:** 業務の流れ、分岐、引き継ぎ、時間依存条件を誤実装させない。
**最低内容:** As-Is、To-Be、開始条件、終了条件、イベント一覧、順序、並列可否、待機条件、分岐条件、差戻し/取消/再処理条件、部門間/システム間引き継ぎ、営業日・休日、締め時刻、月末/四半期末差分、国別/法人別差分。
**元台帳対応:** D
**該当しやすい既存文書名:** 業務フロー、BPMN、業務手順書、運用フロー図。
**不足判定:** To-Be と分岐条件がなければ不足。

### D05. ユースケース・シナリオカタログ

**必須度:** Core
**目的:** 実装優先度と試験対象を利用者目線で整理する。
**最低内容:** 主ユースケース、開始条件、成功結果、正常系、代替系、異常系、利用頻度、ボリューム、高優先、高リスク、監査対象、一括処理、管理者、外部パートナー、遅延/オフライン。
**元台帳対応:** E
**該当しやすい既存文書名:** ユースケース一覧、ユーザーストーリー、シナリオ定義、オペレーション例。
**不足判定:** 正常系だけで異常系がなければ不足。

### D06. 業務ルール・判定表仕様書

**必須度:** Core
**目的:** AI や実装者が推測で条件分岐を作らないようにする。
**最低内容:** ルールID、目的、入力、出力、判定表、計算式、閾値、優先順位、競合時解決順、例外、免除、override 条件/承認者、発効/失効日、国別/商品別/顧客別/契約別差分、説明可能性、根拠規程、変更トリガー、禁止ケース。
**元台帳対応:** F
**該当しやすい既存文書名:** ルール定義書、判定表、料金規程、承認基準表。
**不足判定:** 判定表か計算式のどちらかが欠ければ不足。

### D07. 用語集・ドメインモデル定義書

**必須度:** Core
**目的:** 言葉の意味とモデル境界を固定する。
**最低内容:** 業務用語、定義、同義語、禁止語、国別呼称差分、エンティティ一覧、エンティティ定義、主/外部識別子、ライフサイクル状態、状態遷移、不変条件、集約、所有/参照関係、派生項目、業務イベント、例外事象。
**元台帳対応:** G
**該当しやすい既存文書名:** 用語集、ドメインモデル、状態遷移図、ER前段整理。
**不足判定:** 状態と不変条件がなければ不足。

### D08. データモデル・SoR/SoT・データ品質仕様書

**必須度:** Core
**目的:** 既存データと新実装の接続点を定義する。
**最低内容:** 概念/論理/物理データモデル、主要テーブル/カラム、型、制約、履歴、監査カラム、マスター/トランザクション、時刻/TZ方針、添付メタ、PII/機微分類、データ品質、欠損値意味、名寄せ、保存/削除/アーカイブ、レジデンシ、正常/異常/境界値サンプル、血統、更新主体、SoR、旧新マッピング。
**元台帳対応:** H
**該当しやすい既存文書名:** ER図、DDL、データ定義書、データマッピング表、データ品質基準書。
**不足判定:** SoR、保持/削除条件、PII分類がなければ不足。

### D09. システムコンテキスト・責任境界・再利用方針書

**必須度:** Core
**目的:** 周辺システムとの関係と責任境界を定義する。
**最低内容:** コンテキスト図、上流/下流一覧、外部SaaS、内部共通基盤、SoR/SoE/SoI、責任境界、廃止予定、置換対象、重複機能、バッチ群、常駐/定期ジョブ、障害時代替経路、再利用方針、変更禁止システム。
**元台帳対応:** I
**該当しやすい既存文書名:** システム関連図、責任分界図、連携全体図、現新比較図。
**不足判定:** コンテキスト図と責任境界の両方がなければ不足。

### D10. API / Event / File 連携契約パック

**必須度:** Core（連携がある場合）
**目的:** 連携仕様を契約として固定する。
**最低内容:** API一覧、OpenAPI、イベント一覧、AsyncAPI、エンドポイント、チャネル、リクエスト/レスポンス/メッセージスキーマ、サンプル payload、業務/技術エラー、タイムアウト、retry、idempotency、順序保証、重複配送時挙動、レート制限、SLA、認証/認可、バージョニング、後方互換、破壊的変更、DLQ、再送、ファイル連携、項目マッピング、カットオーバー時二重送信防止。
**元台帳対応:** J
**該当しやすい既存文書名:** API仕様書、OpenAPI、イベント仕様書、IF定義書、CSVレイアウト。
**不足判定:** スキーマとエラー契約がなければ不足。J-16/17/18 は必ず独立して持つ。

### D11. 画面・UX・操作意味仕様書

**必須度:** Conditional
**目的:** 画面が業務上何を意味するかを固定する。
**最低内容:** 画面一覧、画面目的、遷移、ジャーニー、タスクフロー、フィールド定義、入力制約、条件表示/活性、検索/ソート/フィルタ、ページング、一括操作、下書き保存、確認ダイアログ、Undo、エラーメッセージ意味、管理操作、権限差分、監査表示、キーボード/スクリーンリーダー、文言外部化。
**元台帳対応:** K
**該当しやすい既存文書名:** 画面仕様書、Figma、ワイヤーフレーム、UI要件書。
**不足判定:** フィールド定義だけで操作意味がなければ不足。

### D12. 権限・認可・職務分掌設計書

**必須度:** Core
**目的:** 誰が何をできるか、何をしてはいけないかを固定する。
**最低内容:** ロール一覧、階層、権限一覧、操作権限、データスコープ、国/法人/部門/顧客/商品別スコープ、代理権限、一時権限、緊急権限、SoD、承認者/実行者分離、ロール付与元、人事イベント連動、外部委託先権限、監査人権限、再認証、昇格条件、Service Account、API client、テナント分離。
**元台帳対応:** L
**該当しやすい既存文書名:** 権限設計書、認可マトリクス、SoD表、IAM連携仕様。
**不足判定:** SoD とスコープ権限のどちらかが欠ければ不足。

### D13. セキュリティ・プライバシー・監査・法規マトリクス

**必須度:** Core
**目的:** 規制、監査、情報保護の要件を実装制約に落とす。
**最低内容:** 適用法規、社内規程、分類基準、機微データ判定、暗号化、鍵管理、MFA、セッション、認証強度、アクセスログ、監査証跡、改ざん防止、ログ廃棄、異常検知、不正アクセス検知、同意、利用目的、二次利用、本人開示/訂正/削除請求、Export 要件、越境移転、データ主権、委託先共有、高リスク処理追加承認。
**元台帳対応:** M
**該当しやすい既存文書名:** セキュリティ要件書、DPIA、監査要件表、法規対応表。
**不足判定:** 監査証跡、保持、越境移転のいずれかが欠ければ不足。

### D14. 国際化・地域差分仕様書

**必須度:** Conditional
**目的:** 地域差分を「例外」ではなく設計前提にする。
**最低内容:** 対応言語、文言管理、国別文言、通貨、為替処理、税制、日付/時刻形式、タイムゾーン、週開始、祝日、氏名/住所/電話/郵便番号形式、文字種、照合/ソート順、国別法定帳票、本人確認差分、承認差分。
**元台帳対応:** N
**該当しやすい既存文書名:** ローカライズ要件、各国差分表、税制要件、通知翻訳表。
**不足判定:** 税制・TZ・休日差分があるのに文書がなければ不足。

### D15. 非機能・運用・監視・DR 仕様書

**必須度:** Core
**目的:** 性能、信頼性、運用継続性を実装前提にする。
**最低内容:** 可用性、レイテンシ、スループット、同時実行数、バッチ完了時間、データ鮮度、整合性、冪等性、重複防止、再実行性、監視指標、アラート、ダッシュボード、ログ分類、トレース、オンコール、障害レベル、一次切り分け、手動再処理、データ補正、バックアップ、RPO、RTO、フェイルオーバー、復旧試験、保守時間帯、休日運用、Runbook。
**元台帳対応:** O
**該当しやすい既存文書名:** NFR定義、SRE運用設計、監視設計書、DR計画、Runbook。
**不足判定:** SLO と DR 条件がなければ不足。

### D16. 移行・導入・ロールアウト計画書

**必須度:** Conditional
**目的:** 切替・移行・教育・安定化を計画可能にする。
**最低内容:** 移行方針、Big bang/段階移行、地域/法人/顧客群別ロールアウト順、移行対象範囲、移行元/移行先マッピング、データクレンジング、移行ツール、リハーサル、凍結条件、切替判定、移行後照合、ロールバック条件、告知、教育、FAQ、サポート窓口、Hypercare。
**元台帳対応:** P
**該当しやすい既存文書名:** カットオーバー計画、移行計画、展開計画、教育計画。
**不足判定:** 既存システム置換やデータ引継ぎがあるのに照合条件・ロールバックがなければ不足。

### D17. 品質保証・UAT・受入パッケージ

**必須度:** Core
**目的:** 何をもって受け入れるかを固定する。
**最低内容:** 受入基準、業務シナリオ試験、例外系、境界値、データ整合性、権限、監査証跡、多言語、タイムゾーン、レジデンシ、API契約、Event契約、再実行、障害復旧、ロールバック、性能、可用性、アクセシビリティ、セキュリティ受入、法務/監査承認条件。
**元台帳対応:** Q

**不足判定:** 業務試験だけで権限・監査・契約試験がなければ不足。

### D18. Prompt ガバナンス・入力統制パック

**必須度:** Core（Vibe Coding 前提では必須）
**目的:** AI に何を渡してよいか、どの資料を真実源泉とみなすかを統制する。
**最低内容:** Prompt 用 SoT 一覧、資料優先順位、投入可能情報、投入禁止情報、authoritative prompt の保管場所、Prompt pack バージョン、対象リリース/環境/地域/法人/テナント、再利用可能既存モジュール、禁止ライブラリ/禁止SaaS、廃止仕様参照禁止、未解決論点、意思決定ログ。
**元台帳対応:** R
**該当しやすい既存文書名:** Prompt pack、AI入力統制ルール、参照資料優先順位表。
**不足判定:** SoT 一覧、投入可否、廃止仕様禁止の3つがなければ不足。

## 2. 追加した新規文書クラス

### D19. ソフトウェアアーキテクチャ・ADR パック

**必須度:** Core
**目的:** 「何を作るか」ではなく「どういう構造で作るか」を固定する。
**最低内容:**

* 目標アーキテクチャと設計原則
* Solution strategy
* Container 一覧、責務、主要技術選定
* Component 分解、依存禁止ルール
* Runtime シナリオ、正常系/障害系シーケンス
* Deployment view、環境別構成、ネットワーク境界
* データ整合性境界、transaction boundary、cache 方針
* ADR 一覧
* 既知リスク、技術的負債、未解決アーキ論点

**該当しやすい既存文書名:** アーキテクチャ設計書、コンテナ図、コンポーネント図、ADR、技術方針書。
**不足判定:** Context 図しかなく、Container/Component/Runtime/Deployment/ADR がない場合は不足。

### D20. セキュア設計・実装ガードレール

**必須度:** Core
**目的:** セキュリティ要件を設計・実装規約へ落とす。
**最低内容:**

* 脅威モデル
* Abuse case
* trust boundary
* 認証/認可パターン
* 秘密情報管理
* 設定管理
* ログマスキング
* 入力検証/出力エンコード/ファイルアップロード規約
* セッション/CORS/CSRF/署名検証などの規約
* テストデータの匿名化/マスキング
* feature flag / kill switch
* break-glass と監査条件

**該当しやすい既存文書名:** Threat model、secure coding guideline、security architecture note。
**不足判定:** M章の要件はあるが、threat model と実装規約がなければ不足。

### D21. CI/CD・ビルド・リリース・供給網管理仕様書

**必須度:** Core
**目的:** 生成コードを含む開発物の品質と由来を管理する。
**最低内容:**

* リポジトリ戦略
* ブランチ/マージ/レビュー基準
* CI 品質ゲート
* テストゲート
* SAST / DAST / dependency scan
* SBOM
* build provenance / artifact signing
* artifact repository
* IaC / deployment manifest の SoT
* 環境昇格条件
* rollback automation
* 互換性/廃止方針

**該当しやすい既存文書名:** CI設計書、DevSecOps標準、release engineering guideline、supply-chain control doc。
**不足判定:** コード生成を行うのに、CI 品質ゲート・依存スキャン・artifact 管理がなければ不足。

## 3. 非捏造運用ルール

1. Confirmed だけを authoritative prompt に入れる
2. Tentative は Design Assumptions に隔離する
3. Unknown は UNKNOWN のままにする
4. 出典なし情報を Confirmed 扱いしない
5. 資料矛盾時は資料優先順位に従う
6. AI に業務ルールそのものを決めさせない
7. 発効日未定・承認未了ルールを実装前提にしない
8. 例外承認と手動 override は必ず監査ログ化する
9. 本番データを Prompt に渡す場合は投入可否・マスキング規則を先に満たす
10. 生成コードは対応する要求ID・ルールID・ADR へ逆引きできるようにする
11. 未解決論点が残る場合は「仮決め実装」を禁止し、未決として隔離する
12. Prompt pack はリリース単位でバージョン固定し、再現可能にする
