Workflow registry

Step、依存、Agent をデータ定義

Python 側の workflow_registry が順次、並列、AND join を表現し、bash 実装との整合もテストされています。

# hve/workflow_registry.py — StepDef データクラス @dataclass class StepDef: id: str # "1", "1.1", "7.3" title: str # Issue タイトル custom_agent: Optional[str] # "Arch-DataModeling" 等 depends_on: List[str] # AND 依存先 body_template_path: Optional[str] # テンプレートパス skip_fallback_deps: List[str] # スキップ時の代替依存 block_unless: List[str] # ブロック条件
Template-driven

Step ごとの入力文面を統一

各 Step はテンプレートで入力を構成するため、実行チャネルが違っても指示の意味が揺れません。

.github/scripts/templates/ ├─ aas/step-1.md, step-2.md ├─ aad/step-1.md, step-1.1.md … step-7.3.md（12 テンプレート） ├─ asdw/step-1.1.md … step-8.md（24 テンプレート） ├─ abd/step-1.md … step-5.md（9 テンプレート） ├─ abdv/step-1.md … step-4.md（7 テンプレート） └─ akm/step-1.md

Contrast

違いはどこにあるか

Web UI は非同期の自動連鎖に強く、CLI は探索、試行、修正の密度に強いです。両方が必要な理由が、実装に表れています。

Web UI / Batch automation

人の操作をイベントに変える

  • Issue テンプレートで branch、steps、review、qa を収集
  • Label と GitHub Actions でワークフローを起動
  • PR マージ後に依存が解けた Sub Issue へ進む
  • auto-qa / auto-context-review コメントで後続確認を自動投稿
.github/ISSUE_TEMPLATE/app-detail-design.yml .github/workflows/copilot-setup-steps.yml .github/workflows/advance-subissues.yml
CLI / Interactive orchestration

会話をセッションに閉じ込める

  • 1 つの Step 内でメイン処理、QA、Review を同一セッションで連続実行
  • 標準入力から回答を受け取り、そのまま成果物へ反映
  • dry-run、max-parallel、MCP 設定、additional prompt をローカルで制御
  • Issue / PR 作成は必要時だけ有効化できる
# CLI 実行例 python -m hve orchestrate \ --workflow aad \ --auto-qa \ --auto-contents-review \ --max-parallel 5 \ --model claude-opus-4.7 # Issue/PR も作成したい場合 python -m hve orchestrate \ --workflow aad \ --create-issues --create-pr
Engineering insight

両立の鍵は「実行チャネル」と「ワークフローモデル」を分離したこと

Issue、PR、Actions は外側の入出力。workflow_registry と StepRunner は内側の実行モデル。この分離があるため、CLI を増やしても GitHub 方式を壊しません。

A
Input surface

Issue テンプレートか CLI オプションのどちらかで起動条件を与える。

B
Shared orchestration

Workflow DAG と Step 定義で、何をどの順に実行するかを固定する。

C
Output surface

PR コメント、Sub Issue 遷移、またはローカルのコンソール出力として結果を返す。

When Web UI wins

非同期に複数人で回したい、Sub Issue で履歴を残したい、PR 中心で進めたい時。

When CLI wins

ローカルで試しながら進めたい、MCP や追加プロンプトを柔軟に差し込みたい時。

Why both matter

設計探索フェーズと自動連鎖フェーズは別の操作感を必要とするためです。
