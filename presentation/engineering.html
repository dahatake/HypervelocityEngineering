Presentation 02 / Engineering

Prompt だけで勝負しない
三層の Engineering

このリポジトリーでは、Prompt を書く前提、読ませる文脈、検証と安全性のハーネスを別レイヤーとして設計しています。 そのため、Agent の品質を個別の会話テクニックに依存させません。

Prompt EngineeringStructured
Context EngineeringRepository-driven
Harness EngineeringOperationalized
Takeaway

三層がつながっているから、Agent が再現可能になる

自由入力に頼るのではなく、テンプレート、必読ファイル、検証ループで実行品質を固定しています。

Layer 1

Prompt Engineering は役割、出力、禁止事項、分割条件まで含めてテンプレート化

AI Agent 設計ガイドでは、Role、Mission、Inputs、Output requirements、Uncertainty、Procedure、Split mode が明示されています。これは書き味ではなく、失敗パターン対策です。

How it works
役割を先に固定

Senior Architect などの責務を与え、何を決めるべきかを絞ります。

出力形状を固定

見出し順やセクション構造を指定し、出力のばらつきを抑えます。

曖昧性の逃がし先を用意

不明点は Open Questions へ回し、勝手な補完を抑制します。

# .github/agents/Arch-DataModeling.agent.md --- name: Arch-DataModeling description: "ユースケースから全エンティティ・データモデルを生成" tools: ['execute', 'read', 'edit', 'search', 'web', 'todo'] --- ## 1) 入力（必読ソース） - docs/domain-analytics.md - docs/service-list.md - docs/app-list.md ## 2) 成果物（固定） - docs/data-model.md - data/sample-data.json
Anti-hallucination
Role → Mission → Inputs → Output requirements → Uncertainty → Procedure → Split mode

重要なのは、良い文章を書くことではなく、何を根拠にして、どこまで出力してよいかを先に制約していることです。

No fabrication Open Questions Time-box
# AGENTS.md §1) 最優先ルール（抜粋） - 捏造禁止：ID/URL/固有名/数値/事実を根拠なく作らない 不明は `TBD` / `不明（要確認）` と明記する - 推論表記ルール：推論で補完した箇所は `TBD（推論: {根拠}）` + 「この回答はCopilot推論をしたものです。」 - 分割判定：見積合計 > 15分 → 実装禁止、plan.md のみ
Layer 2

Context Engineering は Agent の知能ではなく、入力設計で精度を上げる

AGENTS.md の質問票プロトコル、Custom Agent ごとの必読ファイル定義、APP-ID スコープ、ファイル優先順位の設計が、そのまま Context Engineering の実装です。

# hve/prompts.py — QA 質問票の自動生成プロンプト（実物） QA_PROMPT: str = """あなたはプロジェクトマネージャー兼品質保証担当です。 Issue に記述されたタスクの成果物に対して、確認すべき質問事項を 15〜100個リストアップしてください。 各質問は選択式（Yes/No または A/B/C 等）で回答できるように…""" # hve/prompts.py — 3 観点レビュープロンプト（実物） REVIEW_PROMPT: str = """あなたはシニアアーキテクト兼コードレビュアーです。 1回目: 機能的観点（要件充足性、ロジックの正確性） 2回目: 非機能的観点（パフォーマンス、セキュリティ、保守性） 3回目: プロジェクト横断的観点（他ステップとの整合性） 各観点で問題点を 15〜100個リストアップし…"""
Layer 3

Harness Engineering は生成後の運用設計まで含む

検証ループ、安全ガード、エラーリカバリ、3観点レビューが Agent 行動規約と Skill に落ちています。さらに CLI と CI のテストで、自分たちの実行基盤も継続的に検証しています。

Verification loop

Build → Lint → Test → Security → Diff

コード変更前提の Agent はこの 5 フェーズを通してから PR-ready を判断します。

# AGENTS.md §10.1 — 検証ループ（実コマンド） Phase 1 Build: dotnet build --no-restore Phase 2 Lint: dotnet format --verify-no-changes Phase 3 Test: dotnet test --no-build --verbosity normal Phase 4 Security: grep -rn 'sk-|password=|Bearer ' Phase 5 Diff: git diff --stat

参照: .github/skills/harness-verification-loop/SKILL.md

Safety guard

破壊的操作と秘密情報をパターン検知

rm -rf、git reset --hard、Azure 削除コマンド、ハードコード秘密値などを停止または確認要求に振り分けます。

# AGENTS.md §10.2 — 安全ガード（停止パターン抜粋） | CRITICAL | rm -rf /, DROP TABLE, az group delete | HIGH | git push --force, sk- / password= | MEDIUM | --no-verify, --skip-tests

参照: .github/skills/harness-safety-guard/SKILL.md

Error recovery

失敗時の 3 要素出力を共通化

root cause hint、safe retry instruction、stop condition を統一フォーマットで残します。

# AGENTS.md §10.4 — エラーリカバリ契約 ## エラーリカバリ - root_cause_hint: 何が原因と推定されるか - safe_retry_instruction: 再試行する場合の手順 - stop_condition: これ以上進めない場合の停止宣言 # リトライ上限: 最大 3 回（§1 準拠）

参照: .github/skills/harness-error-recovery/SKILL.md
