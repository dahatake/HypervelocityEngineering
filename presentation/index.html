Presentation 01 / Overview

Agent を主役にした
設計と開発の実験場

RoyalytyService2ndGen は、Agent を単なる補助ツールではなく、工程ごとの責務を持つ実行主体として扱うリポジトリーです。 要件定義から設計、実装、QA までをワークフロー化し、Prompt、Context、Harness を一体で設計しています。

Prompt / Context / Harness を見る CLI とバッチ自動化を比較する
0 共通 DAG で管理されるワークフロー（AAS/AAD/ASDW/ABD/ABDV/AID/AQKM）
0 責務別に分割された Custom Agent 群
0 Build / Lint / Test / Security / Diff の検証ループ
0 Web UI とローカル CLI の両立
Why it matters

Agentic Software Engineering を運用まで落とす

  • Agent に読ませる情報を先に設計する
  • 長いタスクは分割し、依存関係で流す
  • 生成後に QA とレビューを自動挿入する
What is special

単なるコード生成ではなく、工程そのものを Agent 化

Agent の役割、入出力、必要コンテキスト、完了条件、レビュー観点を定義し、設計書とコードをつなぐ流れまで整えている点が特徴です。

01

役割分割が前提

ドメイン分析、データモデル、UI、サービス定義、AI Agent 設計、実装、QA を個別の Agent に分解し、万能 Agent への依存を避けています。

02

DAG で進む

ワークフローはステップ依存を持つ DAG として定義され、完了済みステップだけを起点に次を起動します。順次、並列、AND join が共通モデルです。

Sequential Parallel Join
# hve/workflow_registry.py — AAD ワークフロー（16ステップの一部） StepDef(id="1.1", title="ドメイン分析", custom_agent="Arch-Microservice-DomainAnalytics"), StepDef(id="1.2", title="サービス一覧抽出", custom_agent="Arch-Microservice-ServiceIdentify", depends_on=["1.1"]), StepDef(id="7.1", title="画面定義書", depends_on=["6"]), StepDef(id="7.2", title="マイクロサービス定義書", depends_on=["6"]), # ← 7.1 と並列 StepDef(id="7.3", title="TDDテスト仕様書", depends_on=["6", "7.1", "7.2"]), # ← AND join
03

ドキュメント駆動

前工程で作られた成果物を次工程の一次情報として扱うため、設計と実装が断絶しません。Agent もその前提で指示されています。

docs/domain-analytics.md docs/data-model.md docs/service-catalog.md docs/screen-list.md docs/test-strategy.md docs/services/SVC-*-description.md
Narrative

このリポジトリーが強いのは Prompt 単体ではなく、その周辺設計

良いプロンプトを書くことより、Agent が誤解しない入力文脈、危険操作を防ぐガード、生成結果を検証するハーネスを揃えていることに価値があります。

Core thesis

Agent を賢くする前に、Agent が失敗しにくい舞台を作る

この思想が、AGENTS.md の共通規約、Custom Agent の入力一覧、CLI/Actions のオーケストレーション、QA / Review の追加入力に貫かれています。

├─ AGENTS.md ← 全 Agent の強制ルール（§1〜§10） ├─ .github/copilot-instructions.md ├─ .github/agents/*.agent.md ← 48 Custom Agent 定義 ├─ .github/skills/*/SKILL.md ← 再利用可能な技術 Skill ├─ hve/ ← GitHub Copilot SDK ローカルオーケストレーター │ # 主要ファイルの抜粋 │ ├─ orchestrator.py │ ├─ dag_executor.py │ ├─ runner.py │ ├─ workflow_registry.py │ └─ prompts.py └─ .github/workflows/ ← GitHub Actions 自動化
Flow
1
Issue / CLI で起動

必要な branch、step、review、qa 設定を受け取り、対象ワークフローを決定。

2
Step ごとに Agent 実行

テンプレートと Custom Agent によって役割を固定し、成果物を生成。

3
QA / Review / Advance

PR コメントやローカル入力を使って追加確認とレビューを入れ、依存が解けた次工程へ進む。
