Presentation 04 / Deep Dive

実装の芯は
DAG と Session Orchestration

表面的には Agent が多く、ワークフローも多彩ですが、中心にあるのは workflow_registry、DAGExecutor、StepRunner、GitHub Actions の組み合わせです。

Architecture sentence

Input surface と execution core を分離し、QA / Review を後置する

この一文で、このリポジトリーの実装哲学をかなり正確に説明できます。

Core components

重要コンポーネントの関係

実行核はコンパクトです。workflow_registry が定義、DAGExecutor が順序制御、StepRunner がセッション処理、GitHub API / Actions が外部接続を担当します。

Registry

workflow_registry.py

7 ワークフローの StepDef をデータとして保持し、依存関係と Agent 割り当てを統一管理します。

# hve/workflow_registry.py — 7 ワークフロー定義 AAS = WorkflowDef(id="aas", steps=[…]) # 2 steps AAD = WorkflowDef(id="aad", steps=[…]) # 16 steps ASDW = WorkflowDef(id="asdw", steps=[…]) # 24 steps ABD = WorkflowDef(id="abd", steps=[…]) # 9 steps ABDV = WorkflowDef(id="abdv", steps=[…]) # 7 steps AQKM = WorkflowDef(id="aqkm", steps=[…]) # 1 step
DAG Traversal

WorkflowDef

完了済み・スキップ済みステップ ID を受け取り、次に起動可能なステップを算出します。DAGExecutor がこのメソッドを呼び出して並列数を制御します。

# hve/workflow_registry.py — WorkflowDef.get_next_steps() def get_next_steps(self, completed_step_ids, skipped_step_ids=None): for step in self.steps: deps_satisfied = all( dep in effective_done or dep not in existing_ids for dep in step.depends_on ) if deps_satisfied: result.append(step)
Session

StepRunner

同一セッション内で main task、QA、Review を連続実行し、対話の文脈を維持します。

# hve/runner.py — 1 ステップのフロー ┌──────────────────────────────────────┐ │ CopilotSession（コンテキスト保持） │ │ │ │ Phase 1: send_and_wait(prompt) │ │ → Agent がメインタスク実行 │ │ │ │ Phase 2: send_and_wait(QA_PROMPT) │ │ → 質問票生成 → stdin 回答 → 反映 │ │ │ │ Phase 3: send_and_wait(REVIEW_PROMPT) │ │ → 敵対的レビュー + 最終成果物 │ └──────────────────────────────────────┘
Execution choreography

1 ステップの中で何が起きるか

ローカル CLI では、1 ステップを単発実行するのではなく、同一セッションのまま追加確認とレビューを流し込めるのが特徴です。

1
Main Prompt

対象 Step の本処理を Custom Agent 付きで実行します。

2
QA Prompt

必要なら質問票を自動生成し、ユーザー回答またはデフォルト案を反映します。

3
Review Prompt

敵対的レビューを追加し、成果物を最終版に寄せます。

Bottom line

この設計が示しているもの

Agent の能力差を埋めるためにプロンプトを盛るのではなく、Agent の作業単位、前提文脈、検証ルール、遷移手順をアプリケーション開発の一部として実装しているということです。

一番ワクワクする点

これは「Agent を使った開発」のサンプルではありません。Agent が本当に作業者として参加できるように、開発プロセス側を再設計したリポジトリーです。

Design principle

役割分割、入力制御、状態遷移、検証ループの4点で Agent を支える。

Operational principle

人が一度に全部を判断しなくても、質問票とレビューで後から密度を上げられる。

Engineering principle

実行チャネルを変えても、内部モデルは変えない。だから拡張しやすい。
